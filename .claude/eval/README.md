# 루프 엔지니어링 레이어 (Loop Engineering)

> 측정 가능한 **골든셋**을 만들고, **한 번에 하나(기능 1건)만** 바꿔, **점수가 오를 때만** 반영하며 순환시킨다 — 사람은 선수가 아니라 **코치**로서 루프 **위**에서 승인·최종 QA를 맡는다.

이 레이어는 문서가 **싱크를 유지하면서(정합 축) 점점 완성되도록(완성도 축)** 순환시키는 "공장"이다.

## 두 개의 축

```
정합 축(싱크)   ──  doc-consistency-qa  ──  높음 이격 0 이어야 머지 (HARD GATE, 점수 아님)
완성도 축(완성) ──  doc-eval(골든셋 채점) ──  점수가 오를 때만 머지 (SCORE)
```

- **정합 = 통과/탈락**(기존). **완성도 = 점수**(신규). 둘이 합쳐져 "싱크 유지하며 성숙".

## 자산 (이 디렉토리)

| 파일 | 역할 | 영상 대응 |
|---|---|---|
| `golden-set/*.md` | 산출물별 **예/아니오 기준**(버전 고정) | 골든셋(고정 시험지) |
| `scoreboard.md` | **append-only** 점수 이력 | 측정→다시 측정 |
| `lessons.md` | **append-only** 오답노트(시도·Δ·왜 안 됨·교훈) | 실패 루프 |

**골든셋 불변 규칙**: 시험지를 매번 바꾸면 +N점인지 비교 불가 → 골든셋 수정은 **사람의 의도적 행위**(frontmatter `version` 올림)로만. AI(루프)는 골든셋을 못 고친다.

## 워크플로우

- **`doc-eval`** (체중계 · 선수≠심판): 각 문서를 해당 골든셋으로 채점(채점 에이전트=작성 에이전트와 분리, opus) + `doc-consistency-qa` 정합 게이트 실행 → 총점·항목별 pass/fail을 `scoreboard.md`에 append.
- **`doc-improvement-loop`** (공장 · 1바퀴=1걸음): ① 평가(baseline) → ② 제안(오답노트 읽고 기능 1건/최대 결손 1건) → **[사람 승인 게이트]** → ③ 적용(`feature-docs-pipeline`)+재채점 → ④ 합치기(`score_after > score_before` **그리고** 정합 PASS면 commit, 아니면 `git checkout`으로 버림+오답노트) → **1바퀴 후 멈춰** Δ·keep/revert·다음 제안 보고.

## 머지 조건 (핵심)

```
score_after > score_before  AND  정합 게이트 PASS(높음 이격 0)
  → 둘 다 만족: commit (살린다)
  → 정합 깨짐: 버림 (싱크가 hard 제약)
  → 점수 안 오름: 버림 + 오답노트 (점수 오를 때만 살린다)
```

## 사람의 자리 (루프 "위")

- 루프는 **1바퀴(기능 1건)만 돌고 멈춤** → 사람이 다음 바퀴를 승인.
- 머지 전 **승인 게이트**, 마지막 **QA 칸은 사람**(워크플로우가 100% 못 함 — 영상의 마지막 칸).
- AI=인턴(초안만)·사람=사수: 사람 승인 없이 아무것도 안 바뀐다.

## "기능 단위"의 함정 완화

기능 1건 = 문서 6개 동시 변경(영상: "다섯 개 한꺼번에 바꾸면 원인 못 찾는다"). → `scoreboard.md`는 **문서별 점수**를 쪼개 기록한다. 기능은 (순증이면) 머지하되 **어느 문서가 회귀했는지** 표가 짚어줌 → 다음 오답노트/단일 기준 수정의 타깃. 적용은 기능 단위, **진단은 문서·기준 단위**.

## 실행

```
# 측정만 (체중계)

```
