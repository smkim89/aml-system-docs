# Backoffice Planner (PRD + PPT)

## 0. 정본
- 입력: `docs/software/NN-<svc>-sass.md`(설계) + `docs/design/api·db`(필드·엔드포인트) + `docs/tasks/<svc>/00-overview.md` 먼저 → 번호별 태스크(범위·우선순위).
- 양식 정본: `docs/plan/01-fds-sass-functional-spec.md` + `BO-FDS-SASS-Planning_v1.0.pptx`.
- 아키텍처 사실: `_shared/target-architecture.md`.

## 1. 핵심 원칙
1. **한국어 업무 자연어** — 코드·DSL·enum 토큰·에러코드 미노출(예외: 권한 괄호·API 한 줄). enum은 데이터 항목 표에 괄호 병기.
2. **화면 요소 전수 설명** — 검색 필드·테이블 컬럼·입력(필수 포함)·버튼·탭·배지가 기능 설명에 빠짐없이 대응.
3. **룰/조건 문장형 빌더** — 변수·필드명 금지. ①대상 ②측정 ③기간 ④기준값 ⑤추가조건 ⑥동작 + 자연어 미리보기.
4. **누락 방지** — 메뉴마다 마스터 생성 화면·서브 시나리오 점검.
5. **책임 경계** — 타 서비스 소관은 제외 명시.
6. **탭 시나리오 흐름(연속성)** — 한 메뉴 화면의 탭이 **업무 순서(워크플로)** 면(예 검토 필요→상위 승인→처리 이력, 대기→상신→완료, 후보→제출→이력), 그 탭들을 **시나리오 순서로 연속 슬라이드에 1탭=1슬라이드**로 전개하고 `active`를 슬라이드와 일치시킨다. **모든 슬라이드가 같은 부모 탭 바를 유지**해 "한 화면을 순서대로 보는" 흐름이 되게 한다. 행/버튼 **드릴다운(상세·근거·점수분해)** 은 그 탭 시나리오 **안에 통합**(master-detail)하거나 부모 탭 컨텍스트(breadcrumb·"[검토 필요] 상세")로 표기하고, **부모와 다른 별도 탭 바를 가진 분리 화면으로 만들지 않는다**(흐름 단절 금지). 연속 슬라이드는 "다음 [탭] →"·breadcrumb으로 교차 참조.
   - **멀티탭 상세 화면(데이터 분류 탭 포함)도 동일** — 한 상세 화면이 여러 탭(예 기본 정보 / 배포·온보딩 / 소스 시스템 / 정책팩)을 가지면, **그 탭 바를 단 한 슬라이드에만 얹고 나머지를 탭 없는 분리 화면으로 두지 말 것**. 모든 탭을 **같은 부모 탭 바로 연속 슬라이드(1탭=1슬라이드, active 일치)** 전개하고, 각 탭에 실제 내용(표·패널·카드)을 채운다(빈 라벨 탭 금지). 한 상세 화면의 여러 탭은 **같은 기능 ID**(예 AML-TNT-002)를 공유하고 제목에 탭만 ①②③④로 구분한다. **생성/등록 폼처럼 상세와 무관한 별도 화면**은 그 탭 바를 갖지 않으며(별도 기능 ID), "별도 생성 화면 — 목록 [+신규]에서 진입" 식으로 분리를 명시한다. PRD의 화면 흐름·기능 ID도 PPT와 동일 순서·구성으로 일치시킨다.
   - (단순 데이터/상태 **필터 탭**(예 내 케이스/전체/기한임박/종결, 대기/상신/완료)은 같은 목록의 필터이므로 1슬라이드 유지 — 비대상.)
7. **드릴다운 진입 트리거(화면 ID 간 흐름 단절 금지)** — 목록→상세처럼 **다른 기능 ID 화면으로 진입**하는 드릴다운은: ① 진입 화면 첫 슬라이드 상단에 `entry_banner`로 **"↩ 진입 경로: {소스 화면} {탭}에서 [행 ▶/버튼] 클릭"** 을 명시하고, ② **소스 화면의 표 행/버튼에 `▶` + "→ XXX-002"** 아웃바운드 트리거를 단다(어떤 버튼으로 가는지 불명확 금지). ③ 드릴다운은 가능한 한 **소스 화면 직후 슬라이드**에 배치(설정·무관 화면을 사이에 끼우지 말 것 — 끼면 흐름이 끊긴다). PRD 각 화면에도 진입 경로(트리거)를 기재한다.

## 2. 절차
1. 설계서 정독 → 백오피스 화면 그룹·시나리오 도출(모니터링/마스터생성/목록·상세/운영·결재/감사).
2. 기능정의서 작성: 개요(권한·엔티티·상태머신) + 화면별(표·ASCII 와이어프레임·데이터항목·BR) + 부록(매핑·권한매트릭스·에러코드).
3. PPT 작성: 1=커버 / 2=변경이력 / 3+=화면(좌 75% 와이어프레임 + 우 25% 설명). 기능 ID 전수.
   - **⚠ 와이어프레임은 반드시 실제 도형으로 그린다.** ASCII 박스 문자(┌─┐│└┘)를 모노스페이스 텍스트로 박아넣지 말 것(가시성 붕괴). 시각 정본 = `docs/plan/sample.pptx`.
   - **`.claude/skills/backoffice-planner/wireframe_lib.py`(python-pptx) 컴포넌트로 생성**: `frame(헤더·nav·breadcrumb)·info_panel·tab_chips` + 본문 블록 `filters·kpi_cards·callout·table_block·two_panels·panel_table·form_block·condition_builder` + `entry_banner`(드릴다운 진입 경로 배너)·`history_slide`(변경이력 전폭 표). 팔레트(Ant Design)·좌표는 라이브러리 정본.
   - 화면별 블록 스펙(네비·블록 리스트·info 불릿)을 PRD 화면에서 도출해 라이브러리로 렌더. 카드=큰 숫자, 표=헤더+행(우측 숫자 정렬), 콜아웃=주황 박스, 진입 배너=파란 띠(원칙 7).
   - **변경 이력 슬라이드(2번)는 버전별 한 줄 요약**만 — 긴 정합화 서술을 욱여넣지 말 것(겹침 유발). 상세 근거는 PRD 마크다운 변경 이력에 보존(`history_slide`가 마지막 열을 자동 절단).
4. QA: PDF→이미지(`soffice --convert-to pdf` + `pdftoppm`) 렌더 후 **Read로 시각 검증** — `sample.pptx`와 동등한 도형 가시성인지, 겹침·넘침 없는지. PPT↔기능정의서 표시 용어 일치 확인. 미흡 시 수정·재렌더(최소 1회).
5. 정리: 임시 파일 삭제, 양쪽 변경 이력 동일 기록.
