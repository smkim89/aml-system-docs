# 메뉴 IA 재설계 — 운영/설정 영역 분리 (FDS · AML)

- 작성일: 2026-06-18
- 대상 저장소: `aml-system-docs`
- 영향: 정본 PRD(`docs/plan/01·02-*-functional-spec.md`) 메뉴/NAV 섹션, PPT 생성기(`.claude/skills/backoffice-planner/generate_fds.py`·`generate_aml.py`), 재생성 PPT(FDS v8.0 / AML v9.0)
- 비대상: 화면 내부 콘텐츠(필드·enum·드릴다운·API·DB), bo-web 구현 코드

## 1. 문제 (Problem)

현재 두 콘솔 모두 **설정/셋업 메뉴와 일상 운영 메뉴가 한 평면 리스트에 뒤섞여** 있다.

- FDS 현행 순서: 대시보드 → 고객사 → 커넥터 → 스키마·매핑 → 룰 → 그룹·명단(여기까지 설정) → 탐지결정 → 이벤트 → 액션 → 케이스 → 결재 → 규제보고 → Evidence → 감사. 운영자가 매일 쓰는 탐지결정/이벤트/케이스가 6개 설정 메뉴 아래에 묻힘.
- AML은 더 심각: **한 메뉴 안에 운영 동작과 설정이 혼재**한다. `TM 알림·시나리오`(알림=운영 / 시나리오 빌더=설정), `RA·CDD`(RA 분포·고객위험=운영 / RA 모델·CDD 정책=설정).

운영자(탐지/조사/케이스)와 관리자(룰·시나리오·모델 셋업)의 사용 동선이 메뉴 구조상 분리되지 않아 편의성이 낮다.

## 2. 결정 (Decisions)

1. **2영역 분리**: 모든 메뉴를 `운영(OPERATIONS)`과 `설정(CONFIGURATION)` 두 최상위 영역으로 나눈다. 운영이 위, 설정이 아래.
2. **3단 깊이**: `영역 → 기능그룹 → 메뉴`. 각 영역 안을 의미 단위 기능그룹으로 한 번 더 묶어 댑스를 명확히 한다.
3. **혼재 메뉴 분리**: 운영 동작과 설정이 한 메뉴에 섞인 곳은 솔기 따라 운영 메뉴 / 설정 메뉴로 분리한다(화면 ID는 유지, NAV 소속만 변경).
4. **통계는 운영**: FDS `룰 효과 통계`·AML `통계`는 모두 `운영 › 조사·모니터링`에 둔다(두 콘솔 일관).
5. **화면 보존**: 기존 화면 ID·콘텐츠는 전부 유지. 변경 대상은 순서·소속 기능그룹·NAV 표기뿐.

## 3. 목표 IA (Target Structure)

### 3.1 FDS 콘솔

```
▼ 운영 OPERATIONS
  ▸ 조사·모니터링      플랫폼 대시보드(SFDS-DASH-001/002) · 탐지 결정(SFDS-DEC-001/002/003)
                       · 이벤트 조회(SFDS-EVT-001) · 룰 효과 통계(SFDS-STAT-001)
  ▸ 케이스·처리        케이스 관리(SFDS-CASE-001/002) · 액션 운영(SFDS-ACT-001/002)
  ▸ 거버넌스·보고      결재함(SFDS-APPR-001) · 규제 보고(SFDS-REG-001/002)

▼ 설정 CONFIGURATION
  ▸ 연동·데이터        고객사 관리(SFDS-TNT-001/002/003) · 커넥터 관리(SFDS-CONN-001/002/003)
                       · 스키마·매핑(SFDS-MAP-001/002)
  ▸ 탐지 정책          룰 관리(SFDS-RULE-001~006) · 그룹·명단(SFDS-GRP-001/002/003)
  ▸ 감사·증적          감사 로그(SFDS-AUDIT-001) · Evidence(SFDS-EXP-001)
```

FDS는 주로 순서 재배치 + 2영역 헤더. 혼재 분리는 없음(룰 효과 통계만 룰 관리에서 운영으로 이동).

### 3.2 AML 콘솔

```
▼ 운영 OPERATIONS
  ▸ 조사·모니터링      AML 종합 대시보드(AML-DASH-001) · WLF 검토(AML-WLF-001~004)
                       · TM 알림(AML-TM-001) · 통계(AML-STAT-001)
  ▸ 고객위험·심사      RA 분포·고객위험(AML-RA-001/003) · 고객 프로필(AML-CDD-002)
                       · 고위험 등록부(AML-HRR-001)
  ▸ 케이스·처리        케이스 관리(AML-CASE-001/002) · Travel Rule 예외(AML-TR-001)
  ▸ 거버넌스·보고      규제 보고 STR/CTR(AML-REP-001/002) · 기관 RBA 보고(AML-IRA-001)
                       · 결재 대기함(AML-APR-001)

▼ 설정 CONFIGURATION
  ▸ 연동·데이터        고객사 관리(AML-TNT-001/002/003) · Ingest 카탈로그(AML-ING-001)
                       · 명단 소스·임포트(AML-WL-001/003)
  ▸ 탐지·심사 정책     TM 시나리오 빌더(AML-TM-002) · RA 모델 관리(AML-RA-002)
                       · CDD 체크리스트 정책(AML-CDD-001) · 국가위험 관리(AML-CTRY-001)
                       · Policy Pack(AML-PP-001)
  ▸ 감사·증적·내부통제 내부통제 교육(AML-EDU-001) · 감사·증적(AML-AUD-001)
```

### 3.3 AML 혼재 메뉴 분리 매핑

| 기존 단일 메뉴 | → 운영 | → 설정 |
|---|---|---|
| `TM 알림·시나리오` | TM 알림(TM-001) → 조사·모니터링 | TM 시나리오 빌더(TM-002) → 탐지·심사 정책 |
| `RA·CDD` | RA 분포·고객위험(RA-001/003), 고객 프로필(CDD-002) → 고객위험·심사 | RA 모델 관리(RA-002), CDD 체크리스트 정책(CDD-001) → 탐지·심사 정책 |
| `WLF 검토` / `명단 소스·임포트` | WLF 검토(WLF-001~004) → 조사·모니터링 | 명단 소스·임포트(WL-001/003) → 연동·데이터 |

## 4. 변경 작업 (Scope of Edits)

1. **PRD 정본 메뉴 섹션 교체**
   - `docs/plan/01-fds-sass-functional-spec.md`: NAV/메뉴 구조 섹션(14그룹 평면)을 §3.1 2영역·3단 구조로 교체. 화면 목록 표(§16.1)에 영역/기능그룹 열 추가 및 순서 재정렬.
   - `docs/plan/02-aml-sass-functional-spec.md`: NAV/메뉴 구조 섹션(15그룹 평면)을 §3.2로 교체. 혼재 메뉴 분리(§3.3)를 화면 표에 반영(TM-001/TM-002, RA-001·003/RA-002, CDD-001/CDD-002를 별도 메뉴로 표기).
   - 두 PRD 모두 버전·변경이력 갱신.

2. **PPT 생성기 NAV 정의 수정**
   - `generate_fds.py`·`generate_aml.py`의 NAV 그룹 배열을 `영역(운영/설정) 헤더 + 기능그룹 + 메뉴` 표현으로 변경하고, 각 화면 슬라이드 호출 순서를 §3 순서로 재정렬.
   - 필요 시 `wireframe_lib.py`의 `nav_panel()`이 2영역·기능그룹 헤더(들여쓰기/구분선)를 렌더하도록 소폭 확장(콘텐츠 컬럼은 불변).

3. **PPT 재생성·검증**
   - FDS → `BO-FDS-SASS-Planning_v8.0.pptx`, AML → `BO-AML-SAAS-Planning_v9.0.pptx`.
   - PPT→PDF→이미지 렌더 후 NAV가 운영/설정 2영역·기능그룹으로 표기되는지, 슬라이드 순서가 §3과 일치하는지, 화면 콘텐츠 손실 없는지 확인.

## 5. 수용 기준 (Acceptance)

- 두 PRD의 메뉴 섹션이 운영/설정 2영역 · 3단으로 기술되고, 모든 기존 화면 ID가 정확히 한 곳에 매핑(누락·중복 0).
- AML 혼재 메뉴 3건이 운영/설정으로 분리되어 표기됨.
- PPT FDS v8.0 / AML v9.0이 재생성되고 NAV·순서가 PRD와 일치, 화면 콘텐츠 무손실.
- 통계는 두 콘솔 모두 운영 › 조사·모니터링에 위치.

## 6. 리스크 / 비고

- PPT 생성기는 화면 정의가 Python에 하드코딩되어 있어, 순서 재정렬은 함수 호출 순서 변경으로 처리(콘텐츠 함수 자체는 불변).
- `nav_panel()` 변경 시 기존 슬라이드 레이아웃(좌 75% / 우 25%) 좌표 충돌 없도록 NAV 폭 내에서만 처리.
- bo-web 실제 구현 메뉴와의 정합은 본 작업 범위 밖(문서·PPT 정본 우선 갱신 후 별도 구현 태스크에서 추종).
